# v0.4.3 Maker Setup + Chat-First

Date: 2026-06-24

## Why

Real Maker development testing cannot start until TTMEvolve can prove the boring but critical setup facts: Maker MCP is installed, the active directory is a Maker game project rather than the TTMEvolve app root, TapTap auth exists, remote Maker proxy tools are exposed and registered, and the UI starts from natural-language work instead of a crowded IDE.

## Delivered

- Added Maker Setup Doctor:
  - `GET /maker/setup-status`
  - `GET /maker/setup-status.md`
  - optional `check_latest=true` npm latest-version probe
  - detects configured `@taptap/maker` version, `npx`, `.maker-mcp/config.json`, `.project/settings.json`, project id, auth/PAT files, app-root misuse, and recommended next action.
- Added Maker Tool Audit:
  - `GET /maker/tool-audit`
  - compares remote Maker MCP tools, ToolRegistry registration, Executor handlers, and side-effect proxy marking.
  - required creative proxy tools include image, video, music, and 3D task tools.
- Added project switching/auth preparation:
  - `POST /maker/project/select` switches the active Maker project root and rebuilds the base agent/IDE/Maker MCP runtime when no session is running.
  - `POST /maker/auth/prepare` stores a CLI authorization URL for embedded BrowserView navigation.
  - `POST /maker/auth/complete` checks local auth files and returns `https://maker.taptap.cn/` after success.
- Runtime Contract, Evidence Bundle, and Onboarding Bundle now expose Maker setup and tool audit endpoints/results, so arbitrary LLMs can start from one compact packet before real Maker work.
- Electron exposes a directory picker for Maker project selection.
- Frontend now has a compact Maker Setup strip, an embedded Maker button, a wider chat-first default layout, and Evidence summaries that include setup/audit state.

## Design Notes

The UI direction is chat-native first: conversation is the primary work surface, while setup, provider, Maker browser, files, editor, runtime evidence, and debug tools are supporting surfaces. This follows the common pattern from modern coding agents: keep natural-language intent central, make tool use visible on demand, and move setup/readiness into compact diagnostic strips rather than large permanent panels.

## Current Limits

- TTMEvolve can keep auth inside the embedded BrowserView when it receives the Maker CLI authorization URL. Fully preventing the Maker CLI itself from opening the system default browser depends on CLI support for printing or suppressing auto-open behavior; this still needs a real CLI validation pass.
- `check_latest=true` may be unavailable in offline or blocked-network environments and is deliberately non-blocking.
- Remote creative proxy availability depends on what the connected Maker MCP server exposes; missing remote tools are feature-gated by audit state rather than guessed.
