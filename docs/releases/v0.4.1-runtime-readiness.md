# TTMEvolve v0.4.1 Runtime Readiness

Date: 2026-06-23

## Stable Scope

This is a small stable checkpoint for the API-first Maker cockpit. It focuses on making TTMEvolve easier for any LLM or human operator to enter, diagnose, and continue without replaying raw transcripts.

Completion audit: `docs/releases/v0.4.1-completion-audit.md`

## What Is Stable

- External agents now have a first-stop readiness gate: `GET /runtime/readiness?session_id=...`.
- Runtime Readiness is no-network by default. It reports provider config, API key presence, latest Probe evidence, MakerMCP readiness, layer/context summaries, release-gate checks, and next actions.
- Runtime Readiness and Evidence Bundle include `llm_call_proof`: expected endpoint, observed endpoint, evidence source, and a conclusion such as `api_call_observed` or `api_call_not_observed`.
- Runtime Contract and Quickstart now point external agents to Runtime Readiness before Quickstart/Evidence/Handoff.
- MiniMax/API usage remains evidence-based: use `/llm/probe` or session probe history to prove endpoint, latency, tokens, and errors.
- Saved LLM-as-user feedback is available through `GET /llm/feedback-summary` without sending new project data to an external provider.
- MakerMCP remains the authority surface for Maker remote work. Readiness/Evidence expose connected/tool count/remote identity/last call before detailed `/mcp/tools`.
- Agent/Core Runtime/Learning layer communication is observable through `layer_summary`, runtime metrics, async learning status, Maker guard, and context sync.
- Workbench External Agent Boot includes Runtime Readiness in its copyable boot checklist.
- Workbench visual panels use TapTapMaker semantic tokens and diagnostic text is not clipped.

## Known Limits

- A real API-backed Maker task still needs to be run with the user's configured provider key to verify live latency, first-action Maker alignment, and remote MakerMCP behavior.
- Runtime Readiness does not call providers by itself. This is intentional; use `POST /llm/probe` when a real API check is needed.
- The broader product goal is not complete. This checkpoint gives a stable entry and diagnosis surface for the next real-world validation pass.
